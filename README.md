# Aether Vault: LLM Agent In A 3D Virtual World

This project puts an LLM-style agent into a 3D scene called the **Aether Vault**. The agent perceives the vault through structured observations, chooses from validated actions, and completes a concrete task: recover a brass key, unlock/open a blue force door, enter the emerald beacon, and finish.

The repository includes two pieces:

- A dependency-free Python simulation that owns the actual world state, observation format, action validation, and agent loop.
- A polished Three.js replay viewer that renders the exact JSON trajectory generated by the Python harness.

## Quick Start

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -e .

virtual-world-agent \
  --agent heuristic \
  --log-file examples/demo_heuristic.log \
  --replay-file examples/demo_replay.json
```

Expected result: the run ends with `RESULT: success in 31 steps`.

Run tests:

```bash
python -m unittest discover -s tests
```

## View The 3D Replay

Generate `examples/demo_replay.json` using the quick-start command, then serve the repo root:

```bash
python3 -m http.server 8000
```

Open:

```text
http://localhost:8000/web/
```

The viewer loads the replay JSON, renders the vault, animates the agent path, shows visible entities, and displays the current action/thought/event. It uses Three.js from a CDN, so the browser needs network access for the 3D viewer.

## Run With An LLM

The LLM client uses a generic OpenAI-compatible chat completions API.

```bash
export OPENAI_API_KEY="sk-..."
export LLM_MODEL="gpt-4o-mini"

virtual-world-agent \
  --agent llm \
  --max-steps 80 \
  --log-file examples/llm_run.log \
  --replay-file examples/llm_replay.json
```

For a local or alternative OpenAI-compatible endpoint:

```bash
export LLM_API_BASE="http://localhost:1234/v1"
export LLM_API_KEY="local-key"
export LLM_MODEL="your-model"

virtual-world-agent --agent llm
```

If your endpoint does not support JSON response mode:

```bash
virtual-world-agent --agent llm --no-llm-json-mode
```

## What The Agent Sees

Each turn, the environment returns a structured observation. The LLM never receives hidden world state; it gets a local 3D scan, remembered map, front interaction target, inventory, and valid actions.

```json
{
  "mission": "Explore the Aether Vault, recover the brass key...",
  "world": {
    "name": "Aether Vault",
    "type": "3d_scene",
    "coordinate_frame": {
      "x": "east-west",
      "y": "vertical height",
      "z": "north-south depth",
      "yaw_degrees": "0=north, 90=east, 180=south, 270=west"
    }
  },
  "agent": {
    "grid_position": { "x": 5, "z": 5 },
    "pose": {
      "position_m": { "x": 0.0, "y": 1.45, "z": 4.0 },
      "yaw_degrees": 90,
      "facing": "east"
    },
    "inventory": ["brass_key"]
  },
  "front_cell": {
    "grid_position": { "x": 6, "z": 5 },
    "terrain": "sealed_force_door",
    "passable": false
  },
  "visible_entities": [
    {
      "label": "locked blue force door",
      "distance_m": 2.0,
      "bearing_degrees": 0.0,
      "blocks_movement": true
    }
  ],
  "known_map": "######?????\n#...#...???\n...",
  "available_actions": [
    { "type": "turn_left" },
    { "type": "turn_right" },
    { "type": "look" },
    { "type": "unlock" }
  ]
}
```

The expected model response is a single JSON action:

```json
{
  "thought": "I have the key and the door is directly ahead, so I should unlock it.",
  "action": { "type": "unlock" }
}
```

The harness validates the response against `available_actions`. If a model invents an action or tries something impossible, the world rejects it and reports an invalid-action event.

## Design Choices

The key design choice is separating the **harness** from the **presentation**. Python is the source of truth: it validates actions, updates state, and exports a replay. The browser does not fake the task; it visualizes the exact trajectory the agent actually executed.

Observations are deliberately 3D but compact. The agent gets metric pose, yaw, local depth-scan cells, visible entities with distance/bearing, and a remembered map. This gives the LLM enough spatial context to act without exposing hidden state.

The action space stays physical and inspectable: turn, move forward, look, pick up, unlock, open, finish. That keeps model output grounded in world transitions that can be tested.

The deterministic heuristic policy is included so the system can be verified without an API key. The LLM policy uses the same observation/action contract, which is the main point of the challenge.
